# 验收标准模板 · AC-NNN 六要素

借 ecc intent-driven-development。每条验收标准是一个可验证的契约,不是一句愿望。

## AC-NNN 六要素

每条 AC 编号(AC-001, AC-002…),包含六要素:

| 要素 | 说明 |
| --- | --- |
| 起始条件 | 触发前系统 / 用户处于什么状态 |
| 触发 | 什么动作 / 事件触发这条行为 |
| 预期产出 | 触发后必须发生什么(可观测) |
| 禁止副作用 | 明确不该发生什么 |
| 验证方法 | 怎么客观验证这条达成(命令 / 步骤 / 可观测信号) |
| 优先级 | P0 必须 / P1 应该 / P2 可选 |

**禁空词**:不准用 "correctly / securely / 优雅地 / 快速地" 这类不可验证的词。把它们拆成可测条件。

**具体性铁律(拒绝假大空)**:每条**核心价值 AC**(P0,以及任何"生成 / 推荐 / 展示 / 文案"类)必须附 **≥1 条真实样例**——真实文案 / 真实样例数据 / 真实结果原文,不是占位词("某发票 / X 客户 / 一道菜")。用 `示例:` 起头或代码围栏写出来。一条核心 AC 没有真实样例 = 你还没想清楚它,打回。逆向无真人时,样例标 `[ILLUSTRATIVE-EXAMPLE: … | 待真人替换]`。

## EARS 句式(把行为压成可验证条件)

- 普通:`WHEN <触发> THE SYSTEM SHALL <行为>`
- 条件:`IF <条件> THEN THE SYSTEM SHALL <行为>`
- 状态:`WHILE <状态> THE SYSTEM SHALL <行为>`

例:`IF 用户在 60 秒内校验失败 3 次 THEN 系统 SHALL 锁定该账号 15 分钟,并记录一条审计日志`。

## 一条 AC 的样子

```
### AC-001 · 登录失败锁定 (P0)
- 起始条件:账号处于正常(未锁定)状态
- 触发:同一账号 60 秒内连续 3 次密码校验失败
- 预期产出:账号锁定 15 分钟;返回明确提示与解锁时间;写一条审计日志
- 禁止副作用:不得泄露"密码对/邮箱不存在"的区别
- 验证方法:脚本连发 3 次错误请求,断言第 4 次返回 423 + 锁定时间
- 示例:第 4 次请求返回的真实提示文案 = "Too many attempts. Your account is locked until 14:32 (15 min). Try again then or reset your password." ← 真实文案,不是"返回明确提示"
- 优先级:P0
```

注意示范里多出的 **`示例:` 行**——它把"返回明确提示"这种抽象产出,落成一句用户真会看到的真实文案。每条核心 AC 都该带这样一行。

## 范围边界(spec 必备)
- **Out of scope**:逐条列这版明确不做的。没有这段不算完成。
- **未决项**:不确定写 `[NEEDS CLARIFICATION: 问题]`;反推的写 `[ASSUMPTION: ...]`。
- **阻塞型未决项**:会改变产品方向 / AC 验证口径的(如上线 niche、是否含变现),单列一个 `## BLOCKER` 清单;HARD-GATE 时这些必须清零,普通未决项可带病进下一阶段。

## 验证方法的边界(可验证 vs 禁 HOW)
- 允许:期望值、可观测信号、断言(`net+Σ扣除==gross`、返回 423、字段级错误)。
- 禁止:测试框架 / 技术栈 / 实现方式(那是 tech-spec 的事)。
- 一句话:写"应该得到什么",不写"用什么去测"。

## 优先级配比
- P0 = 不达成产品就不成立的最小集(核心算法 / 核心流程正确性)。
- P1 = 重要但可后置(矩阵覆盖、SEO / GEO、内链)。
- P2 = 纯增强。
- 经验:**P0 宁少勿滥**——P0 太多 = 没有优先级。
