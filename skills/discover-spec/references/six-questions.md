# 逼问清单 · 三层:战略 → 产品细节 → 形态与品牌

借 gstack office-hours 的 YC 六逼问做**第一层(战略)**,但战略层不够——只问战略会得到"假大空" spec。三层都要过:

- **第一层 · 战略(下面 6 问)**:这是不是门生意。按阶段路由:0→1 重点问 1/2/3,1→N 重点问 4/5/6。
- **第二层 · 产品级下钻**:每个核心功能追到"能画出屏 / 能写出一条真实样例"(见末尾「产品级下钻」)。
- **第三层 · 形态与品牌**:这是什么形态、怎么被找到、叫什么、什么声音(见 `product-form-strategy.md` + `brand-system.md`)。

每个问题三段式:**Ask** → **Push-until**(追问到这个程度才算够)→ **Red-flags**(听到这些要警惕)。

## 1 · 需求现实性 Demand Reality
- Ask:有谁现在正为这个问题付钱或付出时间?
- Push-until:能说出具体的人 / 公司 / 场景,不是"大家都需要"。
- Red-flags:"所有人都会用"、"市场很大"、没有任何一个具体用户。

## 2 · 现状替代方案 Status Quo
- Ask:他们今天不用你的东西,是怎么凑合的?
- Push-until:说得出现在的笨办法(Excel / 手动 / 某竞品 / 忍着)。
- Red-flags:"现在没有解决方案"(几乎总是错的,只是你没找到)。

## 3 · 绝望级具体性 Desperate Specificity
- Ask:谁对这个问题已经痛到愿意马上付钱?
- Push-until:锁定一个最痛的细分人群 + 一个最痛的具体场景。
- Red-flags:泛泛的"中小企业 / 年轻人 / 创作者"。

## 4 · 最窄切入 Narrowest Wedge
- Ask:能砍到多小,还依然有人要?
- Push-until:一个一句话能说清、一周能做出的最小切口。
- Red-flags:"平台"、"生态"、"全家桶"、什么都想做。

## 5 · 观察与意外 Observation & Surprise
- Ask:你亲眼看到的、和直觉相反的用户行为是什么?
- Push-until:一个具体的、你没预料到的真实观察。
- Red-flags:全是推测 / 二手转述 / 没真见过用户。

## 6 · 未来契合 Future-Fit
- Ask:为什么是现在?什么变了让它现在成立?
- Push-until:一个具体的趋势 / 技术 / 行为变化。
- Red-flags:"一直都需要"(那为什么之前没人做成?)。

## 第二层 · 产品级下钻(每个核心功能都追到这一层)

战略层答完,**对每个核心功能**继续追,直到能看见真实的产品,而不是抽象描述。Push-until 的终点从"具体场景"上移到"**能画出那一屏 / 能写出一条真实样例**"。

- **第一屏到底显示什么?** 用户进来看到的第一个画面,逐字描述里面有什么字、什么控件。
  - Push-until:能据此画出线框,而不是"一个简洁的首页"。
- **给我一条真实样例,直接写出来,别描述。** 核心功能产出的东西——一条真实文案 / 一行真实样例数据 / 一份真实结果——原样写出。
  - 真实样例 vs 假大空:✅ 把那封催款邮件第 1 轮全文写出来;❌ "生成得体的催款文案"。
  - Push-until:样例具体到能直接拿去用,而不是占位词("某发票 / X 客户")。
- **这个动作点哪个控件、之后屏上变了什么?** 关键交互逐步走一遍。
  - Push-until:输入→控件→可观测变化,三段都具体。
- **Red-flags**:全程只有抽象名词("智能推荐""个性化报告""一键管理"),没有一个真实样例 → 你还没想清楚这个功能,别写进 spec。

## 第三层 · 形态与品牌(按形态路由,不可跳过)

- **这是什么形态?**(内容站 / App / 双边 / 工具 / 社交)→ 按 `product-form-strategy.md` 补齐本形态的强制战略段。
  - **若是 website**:必问 SEO(打哪些真实关键词簇)+ GEO(怎么被 ChatGPT/Perplexity 引用),并写出真实标题/meta/答案块样例。不谈这两段 = 网站类残废。
- **叫什么、对谁、什么声音?** → 按 `brand-system.md` 逼出定位陈述(真实一句)、2–3 个真实命名候选、品牌声音的"品牌腔 vs 通用腔"真实对比。
- **合起来是什么?** 功能咬合成什么系统、给用户什么单一身份/承诺——不是功能清单。
- **Red-flags**:"先把功能做出来,SEO/品牌以后再说" → 对网站类,被找到 = 立项级,不是优化级;品牌定位决定差异化,也不是装饰。

## 反谄媚(贯穿全程)
禁说"这想法很棒 / 有意思"。每个回答都要表态:哪里站不住、哪里有风险、哪个假设最脆弱。

## 逆向版映射(Mode B,无真人可问时)

被当 subagent 自动跑、或纯逆向时,没有真人回答六逼问。规则:把每个逼问转成 `[ASSUMPTION]` 自答 + 标来源,写进 spec 的指定段落,**不要跳过**。

| 逼问 | 逆向时怎么答 | 写进 spec 哪段 |
| --- | --- | --- |
| 谁用 / 需求现实性 | 从产品定位 + 竞品反推目标用户 | 背景 / 问题 |
| 现状替代方案 | 竞品就是现状替代 | ASSUMPTION |
| 为什么是现在 | 从市场 / 技术变化推 | 背景 |
| 最窄切入 | MVP 复刻范围 | 复刻范围段 |
| 明确不做 | 产品没做的 + 你不抄的 | out-of-scope |
| 观察与意外 | 竞品分析里的 commonGap | 差异化点 |

每条标 `[ASSUMPTION: … | 来源: …]`(如 SERP 实测 / teardown §x),不伪装成事实。

**逆向时也不许只堆抽象假设**:没真人答,就主动构造 **≥2 条具体产品样例**(真实样例数据 + 真实文案/界面内容),标 `[ILLUSTRATIVE-EXAMPLE: … | 待真人替换]` 作为锚。第二层、第三层照样要过——没人答不等于可以空。
