# 跨切面决策 · 主动给建议 + 和用户确认

这些决策横切整个产品,**漏一个,后面设计和技术全要返工**。规则:**逐条主动提,给推荐 + 理由,和用户确认**,不替他默默决定,也不静默跳过。spec 的「跨切面决策」段把每条记成:**决定 + 一句理由 + `[已确认]` / `[待确认]`**。

提问方式照"带方案来挑刺":**"X 这块我建议 A,因为 B;你定。"** 不是"你想怎么搞 X?"

## 必过清单(按相关性逐条过)

| 决策 | 何时相关 | 默认推荐(给理由) | 怎么和用户确认 |
| --- | --- | --- | --- |
| **多语言 / i18n** | 面向多地区 / 海外 / 移民人群 | 首版**单语言**,除非核心用户天然跨语言。理由:i18n 早上会拖慢每次迭代;框架层留好钩子(如 next-intl)以后加不难 | "首版我建议只做 [X] 语,文案直接写死;以后要多语言用 next-intl 加,不返工。你的人需要一上来多语言吗?" |
| **平台** | 任何产品 | 先 **web / PWA** 验证,除非强依赖原生能力(推送、相机、后台定位、离线) | "先 web 跑通需求最快;真要原生推送/相机再上 App。你这个有没有非原生不可的能力?" |
| **账号 / 认证** | 要保存数据 / 个性化 / 付费 | **用现成 auth,别自建**;先允许匿名用、价值出现后再引导注册 | "登录我建议用现成服务(见技术段),不自己搭。要不要社交登录?匿名能不能先用?" |
| **支付 / 变现** | 有付费 | 用 Stripe / Paddle / Lemon Squeezy;**做海外 + 怕税务合规优先 Merchant-of-Record(Paddle/LS)** | "收款我建议用 [服务],它替你处理税务/发票。档位怎么定?" |
| **隐私 / 合规** | 收集个人 / 敏感数据 + 面向有法规的市场 | 海外 = GDPR/CCPA:同意 + 导出 + 删除;敏感数据(健康/人格/未成年人)更严 | "你面向 [市场],得做同意+导出+删除;有没有未成年人 / 敏感数据?这条决定要不要年龄门。" |
| **可访问性 a11y** | 公开 web 产品 | 语义 HTML + 键盘可达 + 对比度达标,作底线不作专项 | (基本盘,默认纳入,除非用户明确不要) |
| **数据所有权 / 导出** | UGC / 用户产出内容 | 用户能导出自己的数据 | "用户的内容要不要能导出 / 带走?" |
| **分析 / 度量** | 任何要验证假设的产品 | 埋一套轻量分析对齐战略段的 kill-criteria 指标 | "战略段那几个生死指标,得埋点才测得到——用 [服务]。" |

## 规则

- **相关才提**:纯本地单机工具不必问支付/合规;但凡沾边就得过。
- **每条都落到 spec**:哪怕结论是"不做",也写"多语言:首版不做,理由…,`[已确认]`"——**显式不做 ≠ 漏掉**。
- **`[待确认]` 不是借口**:能给推荐就先给推荐,别把一句"看用户"当交差。
- 这些决策会改技术选型(如多语言→i18n 库、支付→MoR 服务),所以**跨切面 pass 跑在技术 pass 之前**。
