# ADR 模板 · 架构决策记录

借 ecc / Nygard ADR。每个不可逆或高影响的技术决策落一条 ADR,落在 `docs/sdd/<slug>/contracts/` 或 plan 内。
目的:把"为什么这么选"留痕,后人(和未来的 AI)不用重新推一遍。

## 单条 ADR 格式

```markdown
# ADR-001 · <一句话决策标题>

- 状态:提议 / 已采纳 / 已废弃 / 被 ADR-NNN 取代
- 日期:YYYY-MM-DD
- 关联需求:AC-003, AC-007

## 背景
是什么问题逼我们必须做这个决策?约束有哪些(面向地区 / 合规 / 团队技能 / 商业模式 / 成本)?

## 决策
我们选了什么。一句话讲清。

## 候选与权衡
| 候选 | 优 | 劣 | 为什么没选 |
| --- | --- | --- | --- |
| A(选中) | … | … | — |
| B | … | … | … |

## 后果
- 正向:这带来什么。
- 代价 / 风险:接受了什么坏处、埋了什么债。
- 后续:什么条件下要重新审视这个决策。
```

## 什么时候写 ADR
- 选数据库 / 框架 / 部署形态 / 鉴权方式 / 关键第三方
- 任何"以后想改会很贵"的决策
- 不需要为每个小函数写——只为高影响、难逆转的决策写
