# Phase -1 Gates · 动手前的三道门

借 SDD / spec-kit 的 constitution gates。在写 plan、引入任何抽象 / 框架 / 依赖**之前**,必须书面过这三问。
默认拒绝复杂度——过不了门的,不许进 plan。

## Gate 1 · Simplicity(更简单行不行)
- 问:能不能用更少的部件、更少的层做到同样的事?
- 要写:为什么当前方案是"必要的最简",而不是"习惯的复杂"。
- 红旗:"为了扩展性先搭框架"、"以后可能要换所以加一层适配"。

## Gate 2 · Anti-Abstraction(这层抽象现在真需要吗)
- 问:这个抽象 / 接口 / 泛化,是**现在就有两个以上真实用例**,还是"以后可能用到"?
- 规则:只有一个用例 → 不许抽象,直接写具体实现。YAGNI。
- 要写:每个新抽象列出它当下服务的 ≥2 个具体用例,否则砍掉。

## Gate 3 · Integration-First(先打通再优化)
- 问:能不能先做端到端最薄的一条(哪怕丑、哪怕硬编码),验证跑通,再回来优化?
- 规则:先有可运行的垂直切片,再谈分层 / 缓存 / 抽象。
- 要写:第一条 walking skeleton 是什么、怎么验证它通。

## 产出
三道门的 justification 直接写进 `plan.md` 顶部的 "Phase -1 Gates" 段。没写 = plan 不完整。
